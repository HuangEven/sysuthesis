\chapter{综合实验设计、结果分析与讨论}

\section{实验环境与软硬件配置}

本章的作用不是重复展示各章局部结果，而是把前文提出的系统路线放到统一实验框架中检验：融合架构是否在真实或半真实金融数据上保持可用，单卡优化是否确实压缩了数据路径，多 GPU 扩展是否把吞吐提升建立在合理的结构选择之上，以及 I/O 路径与归并方式在何种条件下开始成为新的限制因素\citep{xiao2025flashanns,zhao2022tod}。围绕这些问题，本节先交代实验所依赖的硬件条件、软件栈和实验分类口径。

硬件环境方面，实验平台由多张同型号 GPU、主机 CPU、内存、SSD 以及对应 PCIe 拓扑组成。单卡实验使用 1 张 GPU 运行完整的候选召回与异常评分链路，主要观察第四章提出的数据路径优化、异步流组织、RAPIDS 后端重构和 FAISS-GPU 检索增强在单设备上的综合表现；多卡实验使用 2 张及以上 GPU，主要对应第五章中的 replicated 扩展、控制面调度、GPU 侧归并和 I/O lane 控制。由于本文后续还需要讨论拓扑差异对扩展结果的影响，GPU 型号、显存容量、CPU 型号、主机内存、SSD 容量、PCIe 代际和 NUMA 关系都需要作为实验前提显式给出\citep{gds_overview,gds_design_guide}。

软件环境方面，实验系统由 PyTOD/TOD 评分实现、FlashANNS 候选召回模块、RAPIDS 数据处理组件、FAISS-GPU 检索后端增强组件以及多 GPU 通信和调度运行时共同组成。为了保证结果可复核，本文固定 Python、PyTorch、PyTOD、FAISS-GPU、CUDA、NCCL 以及操作系统版本，并把这些版本视为统一的软件环境快照。由于不同版本的 RAPIDS、FAISS-GPU 和 cuVS 在接口与性能行为上可能存在差异，本章不讨论“哪个版本更优”，而是以当前组合环境作为全文实验的共同基础\citep{cuvs_multi_gpu_nn,nccl_stream_semantics,gds_overview}。

在实验分类上，本文将综合实验划分为三组：方法有效性验证实验，对应真实或半真实数据及 1M 合成桥接口径；单卡系统性能实验，规模口径统一按“真实集 $\rightarrow$ 1M 合成数据 $\rightarrow$ 10M 合成数据 $\rightarrow$ 50M 合成数据”组织；多 GPU 扩展实验，规模口径统一按“真实集 $\rightarrow$ 1M 合成数据 $\rightarrow$ 10M 合成数据 $\rightarrow$ 50M 合成数据 $\rightarrow$ 100M 合成数据”组织，用于比较 replicated 主方案、GPU 侧归并兜底路径以及 I/O 路径组织方式。三类实验共享一套基础代码框架，但在数据规模、硬件使用方式和评价重点上各有分工。

表~\ref{tab:hw-config} 和表~\ref{tab:sw-config} 分别给出了硬件与软件环境快照，图~\ref{fig:exp-topology} 则补充展示了实验平台中 CPU、GPU、SSD 与 PCIe/NUMA 的连接关系，为后续性能差异解释提供硬件背景。

\section{数据集与任务设置}

本章沿用第三章的数据安排，但实验目的已经从“建立系统对象”转向“统一验证系统结论”，因此不同规模数据在这里承担的角色也更明确。真实或半真实金融数据对应全文的现实口径，1M 合成数据承担桥接层，单卡实验沿“真实集 $\rightarrow$ 1M 合成数据 $\rightarrow$ 10M 合成数据 $\rightarrow$ 50M 合成数据”逐层展开，多 GPU 实验则继续延伸到 100M 压力场景\citep{ibm_aml_data,ibm_amlsim,ieee_cis_fraud}。

在有效性验证部分，本文继续使用 IEEE-CIS Fraud Detection、Credit Card Fraud Detection 和 DGraphFin 这三类具有代表性的公开金融数据，分别覆盖结构化支付风控、经典信用卡欺诈以及图金融数据场景\citep{ieee_cis_fraud,dalpozzolo2015creditcard,dgraphfin_paper,dgraphfin_baseline}。这一部分的重点不在于把规模做大，而在于确认融合架构在更接近真实业务分布的数据上仍保持合理的检测效果。

在系统性能与扩展实验部分，本文采用基于 IBM AML / AMLSim 构建的高保真合成金融交易数据\citep{ibm_aml_data,ibm_amlsim}。其中，1M 数据用于连接真实数据验证与系统实验，使检测效果和端到端实现处于同一实验口径；10M 数据是单卡综合性能与多 GPU 扩展的共同起点；50M 数据主要用于观察规模继续增大后单卡链路与多 GPU 扩展的中间趋势；100M 数据则主要服务于多 GPU 扩展、跨卡归并和共享路径冲突实验。这样的规模分层并不是为了把所有规模都做成“同一类实验的重复版本”，而是为了让不同规模分别承担不同验证任务。

在任务组织上，本章实验可分为四类：候选召回与异常评分有效性任务、单卡性能任务、多 GPU replicated 扩展任务以及跨卡归并与 I/O 路径任务。这些任务共享同一套核心实现，但比较对象和观测重点并不相同。有效性任务更关注检测结果与候选排序质量；单卡性能任务更关注数据路径压缩后的吞吐、延迟和利用率；多 GPU 任务则进一步考察 replicated 主方案、GPU merge 兜底路径以及共享路径冲突对扩展收益的影响。

在数据使用方式上，有效性验证任务按训练集、验证集和测试集划分，以保证 PR-AUC、Recall@$k$ 等指标可比；性能与扩展实验则采用固定索引集与固定查询集，将查询批量持续注入系统，以更准确地衡量 QPS、延迟分位数与资源利用率。这样区分的目的，是避免把“检测效果评估”和“系统压测”混在同一套数据口径下讨论。

\section{评价指标与比较原则}

本章采用的指标体系并不是单纯为了“把效果、性能和资源利用率都列出来”，而是为了服务于全文的四条核心判断：融合架构是否在效果上可接受，单卡优化是否主要压缩了数据路径，多 GPU replicated 是否真正提升了吞吐，以及 I/O 路径是否决定扩展上限\citep{aggarwal2017outlier,cuvs_multi_gpu_nn}。围绕这四类判断，指标被组织为四组：异常检测效果指标、候选检索质量指标、系统吞吐与延迟指标以及资源利用率指标。

其中，ROC-AUC、PR-AUC 与 Recall@$k$ 用于回答“候选召回 + 张量化评分”是否仍保持了合理的检测能力；Recall@$k$ 与 Top-$k$ overlap 用于观察候选召回是否改变了最终排序口径；QPS、平均延迟与 p95/p99 延迟直接对应系统在单卡和多卡场景中的端到端收益；GPU 利用率、CPU 占用率、显存使用率、I/O 带宽利用率以及 CPU$\leftrightarrow$GPU 传输开销占比，则用于解释这些收益究竟来自哪里。也就是说，本文并不把“高吞吐”和“检测效果”拆开看，而是始终把指标放回系统路径中理解。

比较原则同样围绕这一逻辑展开。首先，同一类实验尽量共享相同数据集、向量表示和硬件配置，以避免数据口径与平台差异掩盖系统设计差异。其次，在比较不同召回配置时，控制候选集合规模或召回质量处于同一数量级，以保证检测效果与吞吐变化具有可解释性。再次，在多 GPU 实验中固定软件环境与相近查询到达方式，使 replicated、GPU merge、CPU aggregation 和共享路径冲突基线之间的差异尽量来自扩展策略本身，而不是外部扰动。

\section{核心结果展示}

本节不按“图表顺序”逐一陈述结果，而是围绕四个需要被验证的问题展开：融合架构是否总体有效；单卡优化和多 GPU 扩展分别带来了什么收益；多 GPU 主方案与兜底路径的边界如何理解；规模继续增大后系统是否仍保持可用。各小节先给出比较对象和待回答的问题，再引出对应图表。

\subsection{融合系统与基线系统的总体对比}

这一组结果首先回答“融合架构本身是否值得成立”。比较对象包括 PyTOD 全量评分、仅召回方案、初步融合方案以及完整融合方案。这里真正关心的不是某一方案在单一指标上是否最高，而是完整融合是否能够在保持接近最佳检测效果的同时，显著提高吞吐。换言之，若完整融合在 PR-AUC 或 Recall@$k$ 上与全量评分保持接近，却在 QPS 上明显领先，就可以说明“候选召回压缩评分范围 + 候选集上张量化评分”这一结构在系统层面是有意义的\citep{zhao2022tod,xiao2025flashanns}。

\subsection{单卡优化与多 GPU 扩展收益汇总}

第二组结果回答“单卡优化与多 GPU 扩展各自改变了什么”。比较对象包括原始分段链路、完整单卡优化、2 GPU replicated 扩展以及 4 GPU replicated 扩展。这里不把四种方案简单排成“从弱到强”的序列，而是希望区分两类收益来源：单卡优化主要压缩候选召回到异常评分之间的数据路径和尾延迟，多 GPU 扩展则在此前基础上继续抬高系统吞吐上限，并缓解 CPU 压力\citep{faiss_gpu_wiki,rapids_rmm_pinned,xiao2025flashanns,cuvs_multi_gpu_nn}。

\subsection{多 GPU 扩展结果对比}

第三组结果进一步回答“第五章中的结构选择是否成立”。这里的比较对象不再只是 GPU 数量不同的 replicated 方案，还包括在容量受限场景下的 GPU merge 与 CPU aggregation。需要验证的是：replicated 是否确实更适合作为主方案；当主方案不能直接成立时，GPU 侧归并是否优于 CPU 聚合；这些差异是否真正反映在延迟分位数、CPU 利用率和吞吐变化上\citep{cuvs_multi_gpu_nn,nccl_stream_semantics}。

\subsection{不同数据规模下的系统表现对比}

第四组结果回答“规模继续增大之后，系统还能否保持前述判断”。比较对象覆盖 1M、10M、50M、100M 四个规模层级，观察 QPS、p99 延迟和 PR-AUC 如何随规模变化。需要说明的是，PyTOD 全量评分可以作为 1M 量级下的效果上界参考，但并不适合作为 10M 以上规模的统一对照，因此这一组跨规模结果采用“初步融合方案”和“完整融合方案”作为可扩展口径下的比较对象。这里更关注的是，在 out-of-core 路径和多 GPU 扩展逐渐进入关键位置后，系统性能与检测效果是否同时出现可解释的退化：吞吐是否按预期下降，p99 是否继续抬升，PR-AUC 和候选覆盖类指标是否在 10M 之后出现更明显回落，以及 I/O 路径是否开始主导上限。

\section{结果讨论}

从综合结果看，本文提出的四条主结论之间并不是彼此独立的。融合架构之所以成立，并不仅仅因为它“比基线快”，而在于它在效果和吞吐之间建立了更稳定的平衡关系：候选召回没有明显破坏异常评分主干的判别能力，同时又把后续评分范围控制在系统可承受的预算内。对金融异常检测系统来说，这一点比单纯追求某个离线指标更重要，因为它决定了系统是否有机会进入真实高并发场景。

单卡优化的意义也不只是数值上的提速。第四章和本章结果共同表明，系统收益主要来自数据路径被压缩之后，候选召回与异常评分之间的边界成本显著下降，CPU$\leftrightarrow$GPU 传输次数与阶段同步等待被压到更低水平。换言之，原先限制系统的并不是某一个算子绝对不够快，而是模块之间的折返路径过长\citep{faiss_gpu_wiki,rapids_rmm_pinned}。当这一点被解决之后，GPU 侧检索和评分才真正成为决定性能的主要阶段。

多 GPU 结果则进一步说明，第五章给出的结构判断是必要的，而不是形式上的方案区分。replicated 能够成为主方案，并不是因为它在所有条件下都最节省资源，而是因为它最符合吞吐优先场景下的路径组织方式：查询分发简单，单卡链路可以直接复用，CPU 不需要重新承担大规模中间结果处理。当容量约束使 replicated 不再可行时，GPU merge 比 CPU aggregation 更合理，但它仍然只是一条代价更高的兜底路径\citep{cuvs_multi_gpu_nn,nccl_stream_semantics}。

最后，I/O 路径对扩展上限的影响在综合实验中表现得比单卡阶段更直接。随着规模继续增大，系统瓶颈会逐步从计算资源扩展到共享存储路径与拓扑组织上。也就是说，多 GPU 扩展并不是“卡数越多越好”，而是“路径组织能否支撑更多卡同时工作”\citep{gds_overview,gds_design_guide}。这一点也解释了为何本章在扩展收益之外，还必须单独比较 topology-aware binding 与 shared-path conflict 两组结果。

\section{局限性分析}

尽管本文在融合系统、单卡数据路径优化与多 GPU 扩展方面给出了较完整的实验链路，但结论仍需放在明确边界内理解。

首先，超大规模实验主要依赖 IBM AML / AMLSim 等高保真合成数据\citep{ibm_aml_data,ibm_amlsim}。这类数据能够较好支撑系统吞吐、路径压力和扩展性验证，但仍不能完全替代真实金融机构中的在线负载和数据噪声结构。因此，本章关于 10M、50M 和 100M 规模的结论，更适合作为工程可行性的证据，而不是对所有真实业务场景的直接外推。

其次，replicated 架构的优势建立在索引仍可复制的条件上。一旦索引规模继续增大，或候选预算和中间张量占用进一步上升，系统就必须转向 sharded 路径，并接受更高的通信与归并复杂度。这意味着本文对多 GPU 主方案的判断带有明确的适用前提，而不是无条件成立。

再次，out-of-core 路径与拓扑感知 I/O 优化对部署条件依赖较强，其收益与 GPU--SSD--PCIe--NUMA 拓扑、驱动与 GDS 支持高度相关\citep{gds_overview,gds_design_guide}。同一套方法在不同硬件节点上的收益幅度可能存在明显波动，因此本章关于 I/O 路径的结论更强调“方向正确”和“约束真实存在”，而不意味着具体数值在所有平台上都可等比例复现。

最后，本文的实验范围仍然限制在单机多 GPU 场景，多机分布式环境中的跨机通信、分布式索引管理和更复杂的调度问题尚未进入验证范围。这一限制并不影响本文对单机大规模异常检测系统的结论，但确实限定了结论的外推边界。

\section{本章小结}

本章的综合实验把全文前几章分散建立的判断收束到同一验证框架中。结果表明，PyTOD--FlashANNS 融合架构在保持合理检测效果的同时，确实形成了更适合大规模金融异常检测的系统组织方式；单卡优化的主要收益来自候选召回到异常评分之间的数据路径压缩；多 GPU 条件下，replicated 更适合作为吞吐优先场景的主方案，而 GPU merge 与拓扑感知 I/O 控制则主要用于容量受限或路径受限场景下维持扩展效果\citep{xiao2025flashanns,cuvs_multi_gpu_nn,gds_overview}。

这些结论构成了第七章总结与展望的直接基础。下一章将不再重复展开实验过程，而是从系统路线、关键贡献和适用边界三个层面，对全文工作做整体归纳，并进一步讨论后续可继续推进的方向。
